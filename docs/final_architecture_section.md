## MindBloom Flutter + PHP/MySQL Architecture

MindBloom was designed as a real mobile wellness system rather than a static prototype. The solution uses a Flutter frontend for the mobile experience and a PHP/MySQL backend for remote data persistence and cross-session sync. This architecture was chosen because Flutter provides a single codebase for Android and iOS, while PHP/MySQL is widely available on academic hosting environments and easy to demonstrate using phpMyAdmin.

On the client side, the Flutter app follows a modular feature-first structure with Riverpod for state management. Each core module, such as mood tracking, journaling, habits, dashboard, and settings, is separated into its own feature area. Shared services handle notifications, biometric authentication, sensors, geolocation, database access, and backend API communication. This keeps the user interface clean while placing business logic in dedicated services and controllers that are easier to test and extend.

The app uses a hybrid persistence model. Local SQLite storage is used as an offline cache so users can continue interacting with the app even when connectivity is unavailable. A PHP API layer is used for real remote sync to a MySQL database. When the backend URL is configured and reachable, the app sends mood entries, journal entries, habits, and habit completions to the API, which stores them in MySQL. This gives the project a more realistic architecture because it supports both reliability on the device and centralized persistence on the server.

The backend is organized as lightweight REST-style PHP endpoints. `health.php` verifies server connectivity and database access. `bootstrap.php` initializes the remote app user. `moods.php`, `journals.php`, and `habits.php` handle create, read, update, and delete style operations for the main data entities. Journal image uploads are processed by the backend and stored in the uploads directory, while metadata is written to MySQL for later retrieval.

Location and sensor integrations improve the usefulness of the wellness features. Mood and journal entries can optionally capture geographic coordinates, which are then visualized in the app using an OpenStreetMap-based map. This allows the user to identify places associated with calmer or happier moods. Device sensors provide basic motion and inactivity signals, which are surfaced in the dashboard as contextual wellness feedback. Habit and journal reminders are implemented with local notifications so the app remains useful even without a live server push infrastructure.

Security was considered at both the app and backend levels. Sensitive local settings, such as passcodes, are stored using secure storage rather than plain text. Biometric authentication is supported where the device allows it. On the backend, database credentials are intentionally excluded from the public GitHub repository and must be stored only in the server-side `config.php` file. This prevents accidental exposure of the database while still allowing the application to connect securely through the hosted PHP API.

Overall, the selected architecture balances professionalism, realism, and practical implementation scope for a student mobile project. It is advanced enough to demonstrate full-stack integration, modular mobile engineering, and device feature usage, while still being manageable to build, explain, and deploy within an academic setting.
